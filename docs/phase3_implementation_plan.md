# Phase 3 实现计划

## 概述

Phase 3 是网文写作 IDE 的核心功能增强阶段，主要包括功能裁剪、资料管理重写、大纲管理、统计面板、番茄钟和角色关系图等模块。

---

## 核心架构原则

### Skill 系统设计（重要）

**原则**：保持主 AI（Agent）上下文干净，不被污染。

**实现方式**：
1. **番茄预设、写作技能等配置放入 Skill 系统**，不全部加载到上下文
2. **按需调用**：有需要时再调用对应的 Skill，而不是预加载所有内容
3. **不给主 AI 加人设**：主 AI 保持通用助手角色，特定人设通过 Skill 动态注入

**技术方案**：
```
用户请求
    ↓
主 AI (协调者) ← 保持上下文干净，不加人设
    ↓
判断需要什么
    ├── 番茄风格 → 调用对应的Skill（都市赘婿/都市神医等）
    ├── 复杂任务 → 分配给对应的Agent（续写/精修/扩写等）
    └── 资料模板 → 读取数据模板
    ↓
返回结果给用户
```

**Skill vs Agent 设计原则**：

**Skill（技能）**：25个番茄网文风格
- 每个番茄预设是一个独立的Skill
- 包含特定的写作风格和系统提示词
- 按需调用，不污染主AI上下文
- 示例：`skill_tomato_urban_zhuixu`（都市赘婿）、`skill_tomato_urban_shenyi`（都市神医）

**Agent（智能体）**：7个预设智能体
- 每个Agent有独立人设和系统提示词
- 在独立终端窗口运行
- 可以并行处理任务
- 示例：续写Agent、精修Agent、扩写Agent

**分类**：
1. **Skill（25个番茄风格）**：
   - `skill_tomato_urban_zhuixu` - 都市赘婿·隐忍爆发
   - `skill_tomato_urban_shenyi` - 都市神医·一针定乾坤
   - `skill_tomato_urban_zhanshen` - 都市战神·龙帅归来
   - ...（共25个）

2. **Agent（7个预设智能体）**：
   - `agent_continuation` - 续写智能体
   - `agent_refine` - 精修智能体
   - `agent_expand` - 扩写智能体
   - `agent_remove_ai` - 去AI味智能体
   - `agent_outline` - 大纲生成智能体
   - `agent_character` - 角色设计智能体
   - `agent_climax` - 爽点检查智能体

3. **数据模板**（8个资料模板）：
   - 角色模板、设定模板、地点模板等
   - 这些是数据结构，不是Skill也不是Agent

**Agent vs Skill 区别**：
- **Skill**：主AI的能力，如续写/精修/扩写，共享主AI的上下文
- **Agent**：独立智能体，有自己的人设、系统提示词、记忆，如"都市赘婿"Agent

**优势**：
- 主 AI 上下文不被污染，保持通用助手角色
- 每个预设Agent有独立人设，不互相干扰
- 用户可选择不同的Agent获得不同的写作风格
- 可扩展性强（新增Agent不影响主AI）

### 终端多窗口 Agent 架构（推荐）

**核心思想**：其他Agent和主AI形成团队，类似团队协作模式

**区分 Skill 和 Agent**：
- **Skill（技能）**：25个番茄网文风格（都市赘婿、都市神医等），作为写作风格技能，按需调用
- **Agent（智能体）**：7个预设智能体（续写、精修、扩写、去AI味、大纲、角色、爽点），有独立人设，可在终端窗口运行

**架构设计**：
```
主 AI (协调者)
    ├── 番茄风格 → 调用 Skill（25个网文风格技能）
    ├── 复杂任务 → 分配给 Agent（7个智能体，终端窗口运行）
    └── 资料模板 → 读取数据模板

Agent A (续写智能体) ← 终端窗口1
Agent B (精修智能体) ← 终端窗口2
Agent C (扩写智能体) ← 终端窗口3
```

**实现方式**：
1. **主AI作为Team Lead**：负责理解用户需求，分配任务
2. **其他Agent作为Team Member**：在独立的终端会话中运行
3. **终端会话隔离**：每个Agent在独立的终端窗口工作，互不干扰
4. **消息传递协作**：通过消息系统通信，类似现在的团队模式

**技术可行性**：
- ✅ 项目已有终端功能（`StandardTerminalCommandExecutor`）
- ✅ 可以创建多个终端会话
- ✅ 每个会话可以运行独立的Agent
- ✅ 通过消息系统实现Agent间通信

**优势**：
- 主AI上下文完全干净，只负责协调
- 每个Agent在独立环境运行，不会互相污染
- 可以并行处理多个任务
- 用户可以看到每个Agent的工作过程
- 类似现在CodeBuddy的团队协作模式

**示例流程**：
```
用户："帮我用都市赘婿风格续写第5章"

主AI：
1. 创建终端窗口1，启动"都市赘婿"Agent
2. 发送任务："续写第5章，风格要求..."
3. Agent在终端窗口中工作，生成内容
4. 主AI接收结果，返回给用户
```

## 风险与缓解措施

### 风险 1：功能裁剪可能误删依赖代码
**缓解措施**：
- 详细分析代码依赖关系
- 使用工具进行静态分析
- 分阶段删除，逐步验证

### 风险 2：资料管理重写可能引入回归 bug
**缓解措施**：
- 保持向后兼容性
- 完善的单元测试
- 灰度发布策略

### 风险 3：大纲管理功能复杂度高
**缓解措施**：
- 分阶段实现核心功能
- 先实现基础增删改查
- 逐步添加高级功能

### 风险 4：图表库集成可能有性能问题
**缓解措施**：
- 选择轻量级图表库
- 优化数据加载
- 使用虚拟滚动
