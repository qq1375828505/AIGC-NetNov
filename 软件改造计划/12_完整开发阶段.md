# 12. 完整开发阶段与 AI 执行指南

> **目标**：把项目拆成可独立执行、可验证的小阶段，避免一次性写完全部代码后无法调试。  
> **适用范围**：本改造涉及 Android 原生层（Kotlin）+ ToolPkg 插件层（JS/TS）+ HTML 前端三层，必须按依赖顺序分阶段推进。  
> **核心原则**：每一阶段必须先能编译/运行/验证，再进入下一阶段；禁止跨阶段一次性提交大量代码。

---

## 一、Android 项目标准开发流程

```
Phase 0  环境准备 → Phase 1 数据层 → Phase 2 原生框架 → Phase 3 插件骨架
     ↓
Phase 4 核心功能 → Phase 5 AI能力 → Phase 6 辅助系统 → Phase 7 导入导出
     ↓
Phase 8 UI对接 → Phase 9 测试修复 → Phase 10 构建发布
```

## 三、构建策略（本地 vs GitHub）

> 本项目采用 **AI 辅助开发** 模式，构建策略与传统人工开发不同。

### 3.1 核心决策：GitHub 构建为主，本地为辅

| 场景 | 推荐方式 | 原因 |
|------|----------|------|
| **日常代码生成后的验证** | **GitHub Actions** | AI 需要构建日志自动反馈，无需用户手动复制 |
| **阶段性里程碑 / PR 合并前** | **GitHub Actions** | 验证环境一致性，存档构建产物 |
| **Release 发布** | **GitHub Actions 自动化** | 统一签名、生成 changelog、上传分发 |
| **真机调试 / 快速验证小改动** | **本地构建** | 无需等待 CI 排队，直接安装测试 |
| **抓日志 / 分析崩溃** | **本地构建** | 需要 adb logcat、断点调试 |

### 3.2 为什么 AI 开发更适合 GitHub 构建

| 传统开发 | AI 开发 |
|----------|---------|
| 人写代码，编译验证 | AI 生成代码，人审核后编译 |
| 频繁小步迭代 | 每轮可能生成大量代码 |
| 人熟悉环境，出问题能定位 | AI 不了解具体环境配置 |
| 本地调试是日常 | 编译失败时 AI 需要日志反馈 |

**关键原因**：
1. **AI 需要构建日志反馈**：我生成代码后，如果编译失败，需要看到完整的 Gradle 错误日志才能定位问题。GitHub Actions 日志持久保存，本地构建失败时用户需要手动复制日志，效率低。
2. **环境一致性**：我生成的代码可能在本地能编译，但换台机器就不行。GitHub 提供统一的 Ubuntu + JDK + NDK 环境。
3. **版本控制与回滚**：AI 每轮生成代码后 push，GitHub 自动构建。如果某轮代码导致编译失败，可以立刻回滚到上一个成功的 commit。
4. **预编译资源处理**：Operit 的 `models.zip`、`jniLibs.zip` 等大文件可以放在 GitHub Secrets 或 Release Assets 中，CI workflow 中配置下载逻辑，一次配好，后续自动。

### 3.3 GitHub Actions Workflow 配置

创建 `.github/workflows/build.yml`：

```yaml
name: Build AI-Fic-IDE

on:
  push:
    branches: [feature/novel-writing, main]
  pull_request:

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup JDK 17
        uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'
      
      - name: Setup Android SDK
        uses: android-actions/setup-android@v3
      
      - name: Cache Gradle
        uses: actions/cache@v4
        with:
          path: |
            ~/.gradle/caches
            ~/.gradle/wrapper
          key: ${{ runner.os }}-gradle-${{ hashFiles('**/*.gradle*', '**/gradle-wrapper.properties') }}
      
      - name: Download prebuilt resources
        run: |
          # 从 GitHub Release Assets 或私有存储下载
          mkdir -p app/src/main/assets/models
          mkdir -p app/src/main/jniLibs
          wget ${{ secrets.MODELS_URL }} -O models.zip
          wget ${{ secrets.JNILIBS_URL }} -O jniLibs.zip
          wget ${{ secrets.LIBS_URL }} -O libs.zip
          wget ${{ secrets.SUBPACK_URL }} -O subpack.zip
          # 解压到对应目录（根据实际结构调整）
          unzip -q models.zip -d app/src/main/assets/models/
          unzip -q jniLibs.zip -d app/src/main/jniLibs/
          unzip -q libs.zip -d app/libs/
          unzip -q subpack.zip -d app/src/main/assets/subpack/
      
      - name: Build Debug
        run: ./gradlew assembleDebug
      
      - name: Run Unit Tests
        run: ./gradlew test
      
      - name: Upload Debug APK
        uses: actions/upload-artifact@v4
        with:
          name: debug-apk-${{ github.sha }}
          path: app/build/outputs/apk/debug/*.apk
      
      - name: Upload Test Reports
        if: failure()
        uses: actions/upload-artifact@v4
        with:
          name: test-reports-${{ github.sha }}
          path: app/build/reports/
```

### 3.4 本地构建保留场景

| 场景 | 说明 |
|------|------|
| 真机调试 | CI 只能跑模拟器，真机测试必须本地 |
| 快速验证小改动 | 不用等 CI 排队，本地 `gradlew assembleDebug` 几分钟出结果 |
| 抓日志分析崩溃 | 需要 `adb logcat`、Android Studio 断点调试 |
| 离线环境 | 无网络时本地构建兜底 |

### 3.5 AI 开发推荐流程

```
1. 用户告诉我需求 → 我生成代码
2. 用户 review 后 commit + push 到 feature/novel-writing
3. GitHub Actions 自动触发构建
4. 构建成功 → 用户从 Artifacts 下载 APK 真机测试
5. 构建失败 → 用户把 GitHub Actions 日志链接贴给我 → 我分析修复 → 重复 2-4
6. 关键里程碑 → 打 tag → GitHub 自动 release
```

## 四、AI 执行指南

### 4.1 每次只做一个阶段

AI 在接到开发任务时，必须按以下顺序执行：

1. 读取本文件，确认当前应处于哪个阶段。
2. 只实现该阶段内的步骤，不跳到下一阶段。
3. 每完成一个步骤，必须进行验证（编译/运行/测试）。
4. 阶段结束后，向用户汇报阶段成果，申请进入下一阶段。

### 4.2 阶段内的工作流程

```
读取规划 → 创建/修改文件 → 本地验证 → 修复问题 → 提交 commit → 汇报成果
```

### 4.3 Commit 规范

```
[阶段] 模块: 简短描述

示例：
[Phase 1] data: 添加 NovelWork 和 Chapter Room 实体
[Phase 2] ui: 注册 7 个网文写作 Screen 路由
[Phase 4] bridge: 实现 getNovelWorks / createWork
```

### 4.4 遇到阻塞时的处理

| 阻塞类型 | 处理方式 |
|----------|----------|
| 编译失败 | 先修复编译错误，不新增功能 |
| 上游 Operit API 不理解 | 查阅 `docs/package_dev/` 或 `examples/`，不要猜测 |
| UI 设计稿与后端接口不匹配 | 以总纲 NativeBridge 接口为准，必要时调整 HTML |
| 数据库迁移失败 | 回退版本号，修正 migration，不要删用户数据 |
| 跨阶段依赖 | 必须等前一阶段验收通过再做 |

## 五、风险前置

| 阶段 | 常见风险 | 预防 |
|------|----------|------|
| Phase 0 | NDK/Gradle 环境不一致 | 统一用 Android Studio 内置 JDK 和 NDK |
| Phase 1 | 迁移脚本写错导致升级失败 | 写 migration 单元测试 |
| Phase 2 | 导航入口不显示 | 检查 `DrawerContent` 分组逻辑 |
| Phase 3 | ToolPkg 不加载 | 检查 manifest.json、签名、assets 同步 |
| Phase 4 | 编辑器内容丢失 | 自动保存 + 崩溃恢复 |
| Phase 5 | AI 回复不相关 | 必须注入作品上下文 |
| Phase 8 | HTML JS 报错 | 打开 Chrome DevTools 远程调试 |
| Phase 10 | 混淆后 Bridge 失效 | 对 NativeBridge 类 keep |
